P6-T2B - Atomic owner-scoped snapshot sync
Date: 2026-07-14 America/Chicago
Branch: codex/mission/p6-native-offline-outbox

Implemented
- One server transaction now upserts the owner profile and stable snapshot, then appends an immutable audit receipt.
- Separate stable snapshot and logical-operation keys prevent duplicate snapshots and duplicate audit receipts.
- Exact operation replays return the original snapshot ID; changed-payload key reuse fails with SQLSTATE 22023 before mutation.
- A transaction advisory lock serializes concurrent retries for the same owner-operation pair.
- The RPC compares p_expected_owner_id with auth.uid() inside the transaction and fails closed on account changes.
- SECURITY DEFINER is constrained by an empty search_path, fully qualified sensitive objects, JWT-derived ownership, and execute-only authenticated access.
- Authenticated direct profile/snapshot mutations and audit inserts are revoked so clients cannot bypass the atomic contract.
- Null and malformed JSON envelopes are rejected with null-safe validation.
- Web and Expo adapters both use the same RPC and pass no writable owner field.
