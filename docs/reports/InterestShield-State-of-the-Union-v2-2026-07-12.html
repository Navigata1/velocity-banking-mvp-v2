InterestShield / State of the Union v2 / July 12, 2026

The engine is credible. The release and the model contract are not finished.

InterestShield has moved beyond a Replit prototype: it has a shared financial engine, seven web routes, an Expo application, 238 passing regression tests, and a current truth-first dashboard design. The public URL does not serve that product. It serves an older build with offscreen mobile navigation and copy the repository has already corrected. Two core modeling gaps also remain before the app should hold real financial data or present lender-specific projections.

What Survived Adversarial Review

Each high finding was challenged with a second pass using fresh runtime evidence or an independent calculation. No unverified claim drives the verdict.

High / confirmed

The public URL serves the wrong application release.

Local tests and builds pass on GitHub main at dfb4d91, but web-islanddevcrew.vercel.app still serves deployment dpl_FfPyuRhZM8G4pTofYifoajjVDpLg. Browser and Chrome both found the old shell, no primary-navigation marker, no Money Loop artifact rail, and old fear-toned intro copy. At a 375px requested viewport, the browser content width was 360px while the bottom navigation was 492px wide, pushing Wealth Timeline, Settings, and Guardian controls offscreen on every route.

Refutation attempt: seven desktop routes and seven mobile routes were rendered; all loaded without console errors, but all exposed the stale shell. GitHub issue #59 independently tracks the same marker.
High / confirmed

The canonical Money Loop event ledger contradicts its own ending balance.

With the default month, the ledger shows LOC balances of $4,200 -> $0 -> $3,125 -> $4,206.66 -> $3,131.66. Interest should post onto $3,125, producing $3,131.66; instead the interest event adds onto the pre-routing balance, then a second cash-flow event repairs the total. The Simulator renders every event.balanceAfter, so this is visible educational math, not an internal bookkeeping detail. See packages/financial-engine/src/index.ts:531-567 and apps/web/src/app/simulator/page.tsx:454-479.

Refutation attempt: the shared engine was executed with default inputs. Final LOC balance and total interest are correct; only the event sequence is false. Existing tests assert event order and finite values, not ledger continuity.
High / confirmed

The app requires lender terms in its copy but cannot model them in its inputs.

The current copy correctly says users need LOC fees, draw rules, repayment timing, and real terms before trusting a projection. The shared LOCDetails model accepts only limit, APR, and balance; there are no fee, variable-rate, draw-period, minimum-draw, minimum-payment, or repayment-period inputs. With only the three defaults present, the Simulator returns guardrail: null and a 16-month Velocity projection. The CFPB notes that HELOCs can include fees, minimum draws, changing payments, draw periods, repayment periods, and variable rates.

Refutation attempt: repository-wide search found fee and draw-rule language but no corresponding engine field. The default engine snapshot was executed and green-lit the projection.
Medium / confirmed

The Supabase draft has two ownership-contract mismatches.

The migration contract requires owner_id for every collection, including profiles, while public.profiles uses only id. In addition, simulation_runs(snapshot_id, owner_id) does not enforce that the referenced snapshot belongs to the same owner. Fix the contract or schema, then add a composite unique key and composite foreign key before applying the draft. See backend-migration-contract.ts:34-44 and 202607020001_first_lane_owner_scoped_schema.sql:10-37.

Medium / confirmed

The LOC model is an estimate, not a transaction calendar.

ADB uses a 30-day default, deposits all income first, and spreads expenses evenly. This is well labeled in many screens, but it cannot reproduce real paydays, statement periods, bill dates, rate changes, or lender posting conventions. It is suitable for education and sensitivity analysis, not a lender-statement promise.

Medium / confirmed

The visual system is expressive but not yet a real 3D experience.

The current artifact rail uses CSS perspective, conic gradients, SVG paths, and PNG heroes. There is no Three.js, React Three Fiber, Remotion, HyperFrames, GSAP, GLTF, or WebGL runtime. That is a useful prototype, but it cannot deliver the requested selectable rendered object stage. Adding 3D without first shrinking the existing 79.8 MiB image library would create a performance problem.

PASS: source, test, and recomputation agree

Refuted concern: APR units

Whole-percent and decimal inputs are normalized, and regression coverage checks portfolio display and payoff math. No active percent-vs-decimal defect was reproduced.

Refuted concern: Velocity is always fastest

The default shared simulator reports Snowball/Avalanche at 13 months and Velocity at 16 months. The current code does not force Velocity to win, which is the right truth-first behavior.

The 10x Experience Has Distinct Jobs

Three.js, algorithmic art, HyperFrames, and Remotion should not compete inside one runtime. Each gets a clear boundary.

Runtime

Three.js show stage

Build one lazy-loaded React Three Fiber stage on the dashboard. Five selectable artifacts represent Income, LOC, Expenses, Cash Flow, and Principal. Selection triggers one controlled rotation and settles. Real engine values drive material fill, orbit pressure, and warning state. DOM vitals remain outside the canvas.

Generated media

Remotion explainers

Render deterministic scenario videos from versioned engine JSON: baseline vs modeled plan, the first Money Loop month, and why a projection was blocked. Videos become shareable education and onboarding, not a dependency required to calculate a result.

Motion system

HyperFrames campaign assets

Use branded HTML/GSAP compositions for launch films, tutorial chapters, captions, and data-in-motion cutdowns. Keep its validation, contrast, overflow, and deterministic-timeline gates in a separate media workspace.

Algorithmic art rule

Every visual parameter must explain a modeled state: utilization changes shield aperture, cash flow changes pulse cadence, interest burn changes heat, and projection confidence changes focus. Decoration that cannot be traced to data stays out.

Performance and access rule

Load 3D after the four vitals, cap initial compressed GLB assets, pause offscreen rendering, honor reduced motion, provide a static PNG fallback, keep all controls in accessible DOM, and gate desktop/mobile screenshots plus nonblank canvas-pixel checks.

Plan of Attack

The sequence reduces financial and release risk before investing in spectacle. Estimates are focused engineering sessions, not calendar promises.

1

Calculation contract - repair the ledger and model the terms that matter

4-7 sessions / required before real-data launch

  • Make event balances continuous and add a ledger invariant: previous balance plus signed event amount equals next balance.
  • Add transaction-calendar inputs, actual month length, payday and bill dates, rate mode, fees, draw period, repayment period, minimum draw, minimum payment, and optional lender posting rules.
  • Version assumptions and engine outputs. Add golden fixtures from anonymized lender statements and an independent reference-calculator test lane.
2

Supabase foundation - private, recoverable, deletable data

5-8 sessions / starts after schema review

  • Create a dedicated project, update the migration for ownership consistency, run advisors, and test every RLS policy as owner A, owner B, and anonymous.
  • Wire web and Expo auth, snapshot sync, simulation history, learning progress, export metadata, account deletion, and local-demo migration idempotency.
  • Define retention, backups, restore drill, incident logging, and secret handling before inviting real users.

Site and dashboard overhaul - quiet clarity first

6-10 sessions / can overlap backend after contracts freeze

  • Keep the first screen to the four vitals, then stage the Money Loop artifact selector and evidence drawer below it.
  • Break the 1,284-line Learn page, 931-line Portfolio page, and 1,128-line mobile shell into bounded feature modules.
  • Replace emoji navigation with icon components, improve metadata and share previews, compress/convert PNGs, and set route-level performance budgets.

Immersive layer - one real 3D artifact stage

6-9 sessions / performance-gated

  • Prototype the five-artifact selector with React Three Fiber and optimized GLB assets.
  • Bind visuals to versioned engine outputs, implement one-spin-and-settle motion, touch/keyboard parity, static fallback, and reduced motion.
  • Gate with desktop/mobile screenshots, canvas pixel checks, interaction tests, memory checks, and Core Web Vitals.

Media pipeline - HyperFrames and Remotion as publishing tools

  • Publish rendered assets through the app and campaign surfaces without importing either renderer into the core calculator bundle.

Expo hardening - turn the existing port into a native product

6-10 sessions / follows backend and engine contracts

  • Use Expo Go first; preserve shared engine parity and build native route components instead of growing the monolithic shell.
  • Run the configured Android emulator smoke, establish a macOS iOS simulator/CI lane, and test SecureStore, deep links, offline recovery, and accessibility.
  • Only add custom development builds when a native dependency actually requires them.
